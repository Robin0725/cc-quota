# CC Quota 错题集

排障前先查同类案例;修完必记:症状 → 真凶 → 修法 → 教训。

## #001 面板内容无声消失(×2)

- **症状**:面板卡片内容整块不见,无报错。
- **真凶**:一次是 `slice(0, 2)` 截断;一次是 CSS 行高压缩 + `overflow: hidden` 裁掉 Claude 的 Fable 分桶行。
- **修法**:`grid-auto-rows` 保 `min-content` 下限;不加 `overflow: hidden`。
- **教训**:这两道安全网是防静默丢数据的,改面板布局不得拆。

## #002 Kimi 卡片每刻钟整个消失

- **症状**:Kimi CLI 闲置时,菜单栏 Kimi 胶囊每 ~15 分钟消失一次。
- **真凶**:`merge_snapshots` 按 status 把 `signed_out` 当"永久失败"清空读数;而 Kimi token 只活 15 分钟,闲置过期是常态。
- **修法**:只有 `ok` 才替换,其余一切失败退回 30 分钟内上次读数并标 `stale`(合同附录)。
- **教训**:不得按 status 猜"失败是否永久";保留上次读数由 `MAX_STALE_SECONDS` 界定。

## #003 悬浮窗被后台 agent 劫持,固定显示 kimi(2026-07-19,v0.5.2 修)

- **症状**:用户在 Claude 里打字,悬浮球却固定显示 kimi 不切换。
- **真凶**:活动信号用"session 目录最近写入"。一个后台跑长任务的 Kimi agent 每一两分钟流式写盘,"最近写盘者"几乎永远是它。信号测的是 **agent 在干活**,不是**用户在用谁**。
- **修法**:信号改为只有用户亲自输入才变动的提示历史(claude → `~/.claude/history.jsonl`,kimi → `~/.kimi-code/user-history/`);codex 无等价文件,留 sessions,已知局限(合同 §6.1b)。
- **教训**:选"用户正在用谁"的信号时,先问"后台 agent 会不会也触发它"。凡 agent 也会写的路径都不合格。
- **附带发现**:当时线上跑的是 03:53 的 debug 构建(`/Applications` 里也是 debug 包),不含后续修复——发版后要确认用户机器上换成了 release 产物。

## #004 全仓评审三连(2026-07-19,v0.5.2 修)

1. 前端 backoff 把"任一 provider 非 ok"计为失败 → Kimi 闲置(常态)导致轮询钉死在 30 分钟顶格、临近重置的 60s 快路径永不生效。改为"有人成功即健康"。
2. keychain 存在性探测(`security` 子进程)无超时、在 750ms 轮询链路上、不受 PROVIDER_FETCH_BUDGET 覆盖 → 锁定的 keychain 可无限挂起。加 2 秒上限,超时杀子进程按"不存在"处理。
3. 冷启动窗口(取数进行中 + 缓存为空)碰撞请求用 `providers::all()` 造出未登录 provider 的幻影卡。改 `active()`。

## #005 Kimi 胶囊闲置半小时后还是消失(2026-07-19,v0.5.3 修)

- **症状**:#002 修完后,Kimi 闲置超过 30 分钟胶囊仍然消失;悬浮球此时若活动信号指向 kimi,会退到注册表第一位显示 Codex 的数字(CX),像是"切错了"。
- **真凶**:`MAX_STALE_SECONDS` 只保留旧读数 30 分钟,而 Kimi 的 token 要等用户回来用 CLI 才续期——离开吃顿饭就超窗。悬浮球的 CX 是前端 `available[0]` 兜底,不是切换逻辑坏了。
- **修法**:保留窗口改 24 小时(用户拍板),全程置灰标 `stale`;24 小时无成功读数才进空状态。两侧常量 lib.rs `MAX_STALE_SECONDS` / format.ts `MAX_STALE_MS` 必须同改。
- **教训**:定"保留旧数据多久"要按**数据源自己的续命节奏**算,不是拍一个通用分钟数;排查"显示了错的 provider"先看是不是可显示列表为空导致的兜底,别急着怀疑切换逻辑。

## #006 辅助功能权限开了却不生效(2026-07-19,当日修通)

- **症状**:用户在系统设置里打开了 CC Quota 的辅助功能开关(TCC auth_value=2),重启应用后窗口标题识别(点击跟随第一层)仍不工作。
- **真凶(两层)**:①TCC 授权绑定**授权那一刻记录的 cdhash**,换过安装包就对不上;②更隐蔽的一层:CI 的 universal 产物 **lipo 后只有 arm64 半边有链接器 adhoc 签名,x86_64 半边完全没签**——TCC 记录指纹时录的是未签名 x86_64 半边的两个候选 hash,而实际运行的是 arm64 半边(cdhash 完全不同),**无论怎么重新授权都永远对不上**。用 `csreq -r req.bin -t` 解码 TCC.db 的 csreq、对比 `codesign -dvvv --arch` 各架构 CDHash 实锤。
- **修法**:装好后 `codesign --force --deep --sign - "/Applications/CC Quota.app"` 把两个架构都签上 → `tccutil reset Accessibility app.ccquota.desktop` → 由应用自己再发起请求(托盘菜单项,可用 AppleScript 代点)→ 用户在系统设置开关。CI 已加签名步骤防再产半签包。
- **坑中坑**:`tccutil reset` 会把条目从系统设置列表**整个移除**;若用户的设置窗口还开着,看到的是幽灵条目,拨了也白拨。重授权前先关掉系统设置窗口。
- **教训**:①adhoc 签名 cdhash 每次构建都变 → **每次更新版本,辅助功能授权必失效**,装机流程固定附带"补签名 + reset + 重新授权";②排查 TCC"开了没用":解码 csreq 对比各架构 CDHash,注意 universal 包可能半签;③根治=固定自签证书(designated requirement 稳定,授权跨版本存活),待办。

## #007 点悬浮窗展开详情,显示先跳到别家(2026-07-19,v0.5.4 修)

- **症状**:在 codex 窗口工作(悬浮球 CX),点悬浮球展开详情的瞬间,球跳成 KM。
- **真凶**:点击悬浮球让 CC Quota 自己成为前台应用;点击跟随层对"自己"返回 None 落到第 2 层(提示历史=kimi)。"返回 None"的本意是"别动",实际效果是"落层改判"。
- **修法**:`Focus` 三值枚举(Widget/Provider/Unknown)+ 记住上一次答案;前台是自己 → 直接返回上次答案(冻结),只有从未有过答案才照常走层(契约 §6.3 例外条款)。
- **教训**:"不参与判定"和"维持现状"是两回事——在多层兜底体系里,返回"无意见"会把判定权交给下一层,想要"别动"必须显式记住并返回上一次的结果。
